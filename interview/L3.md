# L3 · 状态管理

> 共 30 题。先自答，再展开「参考答案」。链回 `docs/`/`ai/`/`projects/` 仅作延伸，不替代本题答案正文。

62. HashMap、RocksDB、ForSt 三种 state backend 的选型矩阵？

**参考答案：** 答案：HashMap：小状态、低延迟、全在堆，不适合 GB 级。RocksDB：大状态落盘+托管内存，增量 checkpoint，生产默认候选。ForSt：新一代/学习仓库中按版本矩阵评估的大状态选项，需按 SSOT 登记版本再启用。
判据：状态体积、TTL、checkpoint 时长、运维复杂度。见 docs/03-state。

63. Keyed State 与 Operator State 区别？

**参考答案：** 答案：Keyed 随 key 分区，rescale 按 key group 迁移；Operator State（如 source offset 列表）按并行实例重分配，需定义 redistribute 语义。Kafka source 位移多为 operator state。见 docs/03-state。

64. ValueState / ListState / MapState 选型？

**参考答案：** 答案：单值用 Value；追加元素用 List（注意长度）；大量子 key 用 MapState 避免大 List 每次反序列化。热点 key 的 MapState 仍可能成为瓶颈。

65. 状态 TTL 的语义？过期后读取返回什么？

**参考答案：** 答案：可为状态条目设置 TTL，过期后访问视作不存在并清理（懒/后台策略依配置）。错误 TTL 会「丢维表/丢会话」。SQL 用 `table.exec.state.ttl`。见 docs/03-state、docs/05-sql。

66. 为什么说「状态是流计算的本质」？

**参考答案：** 答案：无状态 map 可重放；聚合、去重、join、CEP 都依赖跨事件记忆。容错即「把状态与输入位置一致快照」。见 docs/00-landscape。

67. Broadcast State 的读写约束？

**参考答案：** 答案：广播流写入 BroadcastState；主流只读。用于动态规则（p03 Pattern 切换）。规则集需有界，否则 TM 内存爆。见 docs/10-cep、p03。

68. Queryable State 为何本仓库不作为主路径？

**参考答案：** 答案：运维复杂、一致性语义易误解；学习工程优先把结果写入 Kafka/CH 供查询。见架构约束。

69. 状态序列化用 Kryo 的风险？

**参考答案：** 答案：跨版本不稳定、调试难。优先 TypeInformation / POJO / Avro 等明确 schema。升级作业要验证 serializer 兼容。

70. RocksDB 内存相关参数调优入口？

**参考答案：** 答案：managed memory fraction、block cache、write buffer 等；先保证 managed memory 足够，再看 checkpoint 时长与 thrashing。见 docs/03-state。

71. 什么是 key group？和并行度上限关系？

**参考答案：** 答案：key 空间分成固定 key group；并行实例各持有一组。maxParallelism 决定组数，从而限制可扩最大并行度。

72. 本地状态与 checkpoint 存储的区别？

**参考答案：** 答案：本地（堆/RocksDB 目录）供运行读写；checkpoint 是持久化快照到远程（本仓库 MinIO/S3 兼容）。丢失本地盘可从 checkpoint 拉起。

73. 状态倾斜如何表现？

**参考答案：** 答案：少数 key 状态巨大/访问极热，单 subtask RocksDB 变慢、checkpoint 被拖长。解法：拆 key、局部聚合、两阶段、业务限流。见 best-practice、docs/13。

74. Heap 状态 + 大窗口的致命组合？

**参考答案：** 答案：窗口缓存元素在堆上易 OOM；应增量聚合或 RocksDB + 聚合函数。

75. MapState.clear() 与逐 key 删除的成本？

**参考答案：** 答案：整表 clear 语义清晰；大量逐条删除可能放大 RocksDB compaction。TTL 批量过期更可取。

76. State Descriptor 命名冲突会怎样？

**参考答案：** 答案：同算子内名字必须唯一；uid 变化会导致「找不到状态」走新状态。这是强制 `.uid()` 的原因之一。见 docs/04-checkpoint。

77. UnionListState 用于什么？

**参考答案：** 答案：Operator state 在 rescale 时把列表 union 到各实例再自行拆分——适合 source 拆分列表。

78. CheckpointedFunction 接口何时实现？

**参考答案：** 答案：需要在 snapshotState/initializeState 管理 operator state 时（自定义 source/sink）。普通 Keyed 业务少直接实现。

79. 状态迁移跨版本的检查清单？

**参考答案：** 答案：uid 稳定、serializer 兼容、maxParallelism 不变或可迁移、savepoint 类型匹配、先在 staging 从 savepoint 启动验证。见 production bluegreen SOP。

80. TTL 与窗口状态如何协调？

**参考答案：** 答案：窗口有自己的清理；TTL 用于任意 keyed state。勿对仍需迟到更新的窗口状态设过短 TTL。

81. 如何观测状态大小？

**参考答案：** 答案：checkpoint size 指标、RocksDB native 度量、业务侧估计 key 基数×每 key 字节。Grafana job-deepdive 含 lastCheckpointSize。见 monitoring。

82. RocksDB 目录放空写层容器的后果？

**参考答案：** 答案：重启丢本地态，完全依赖远程 CP，恢复变慢。

83. 状态描述符更换类型后能否兼容？

**参考答案：** 答案：通常不能；需迁移作业或弃状态。

84. ListState 无限追加的事故模式？

**参考答案：** 答案：每事件 add 不清理→OOM；应窗口聚合或 TTL。

85. Broadcast 规则版本字段如何设计？

**参考答案：** 答案：单调 version，实例忽略旧版；与 p03 pattern_id 类似。

86. KeyedProcessFunction 里 timer 服务语义？

**参考答案：** 答案：事件时间定时器随 watermark 触发；处理时间随墙钟。

87. 状态后端切换是否要重新积累状态？

**参考答案：** 答案：通常需兼容迁移路径；随意切换等同放弃旧状态。

88. 大 MapState 迭代的性能陷阱？

**参考答案：** 答案：全量 entries() 扫描昂贵；尽量点查与 TTL。

89. OperatorState 重分布失败的表现？

**参考答案：** 答案：源拆分错乱、重复或空洞消费；需测 rescale。

90. 状态访问线程模型？

**参考答案：** 答案：单 subtask 单线程处理，状态无并发；Async 回调需注意。

91. 状态迁移时并行度从 2→3 的关键？

**参考答案：** 答案：key group 重分配；maxParallelism 必须允许。
