# L1 · 运行时与拓扑

> 共 31 题。先自答，再展开「参考答案」。链回 `docs/`/`ai/`/`projects/` 仅作延伸，不替代本题答案正文。

1. 一个 Flink 作业从 env.execute() 到 Task 跑起来经历了哪几层图转换？各层分别在哪里生成？

**参考答案：** 答案：典型路径为 StreamGraph → JobGraph → ExecutionGraph → 物理 Task。
1) StreamGraph：客户端/本地环境在 DataStream API 调用时逐步构建（算子、边、并行度提示）。
2) JobGraph：客户端提交前由 StreamGraphGenerator / Optimizer 生成，算子链（chaining）在此折叠，形成 JobVertex。
3) ExecutionGraph：JobManager 收到 JobGraph 后展开为并行 ExecutionVertex/ExecutionEdge，做调度单元。
4) 物理 Task：TaskManager 收到部署后在 Slot 内跑 Task 线程。
考点推导：简历口述应能指出「链合并发生在 JobGraph，并行展开发生在 ExecutionGraph」。见 docs/01-runtime。

2. Slot 与并行度的关系？slot sharing 为什么能让一个 slot 跑整条 pipeline？

**参考答案：** 答案：并行度决定每个算子实例数；Slot 是 TaskManager 提供的资源配额。默认 slot sharing group 下，同一 Job 中不同算子的子任务只要属于同一 sharing group，就可共享一个 Slot（共处一条 pipeline 的上下游）。
理由：Flink 认为同一 Slot 内上下游用本地缓冲交换，节省 Slot 数量——所需 Slot 数 ≈ 最大并行度（在单一 sharing group 时），而非所有算子并行度之和。
反例：把重 CPU 算子单独 setSlotSharingGroup 后，Slot 需求上升；资源不足会部署失败。见 docs/01-runtime。

3. 算子链的合并条件有哪些？什么时候你会手动 disableChaining？

**参考答案：** 答案：合并条件（同时满足）：相同并行度、同一 slot sharing group、前后算子可链式（forward 局部传输）、未被禁用、算子类型允许（如部分 co-process 边界）。
手动 disableChaining / startNewChain 场景：需要独立反压观测、重算子避免与轻算子共线程抢 CPU、或要插入旁路监控。
考点：链式是性能默认优化，但排障时拆链能让 busyTime/backpressure 指标落到具体算子。见 docs/01-runtime。

4. TaskManager 与 JobManager 的职责边界是什么？HA 下 JM 故障作业为何能续跑？

**参考答案：** 答案：JM 负责任务调度、checkpoint 协调、作业生命周期；TM 负责任务执行、本地状态/网络缓冲。HA 下多个 JM 候选 + 高可用存储（ZooKeeper/K8s 等）保存 JobGraph 与元数据；活跃 JM 挂掉后 standby 接管，从最近成功 checkpoint 恢复。
推导：无 checkpoint 时 HA 只能「重新提交」而非「续跑状态」。本仓库 compose 基座偏演示 Session；K8s Operator Application 模式见 production/。

5. Application Mode 与 Session Mode 的核心差异？学习工程为何两者都要会？

**参考答案：** 答案：Session：共享 JM/集群，多作业提交，隔离弱、适合开发。Application：一作业一集群（per-job JM），生命周期与作业绑定，生产隔离更好。
学习工程：docker compose Session 便于快速试 e0x；production/ Operator 路径演示 Application/Blue-Green。选型判据是隔离与运维成本，不是「谁更新」。见 docs/14-production、production/docs。

6. Forward / Hash / Rescale / Broadcast 分区各自何时使用？

**参考答案：** 答案：Forward：上下游同并行度本地传递（链内常见）。Hash/KeyBy：按 key 分组保证同 key 进同实例。Rescale：上下游并行度不同时的轮询式局部重平衡。Broadcast：规则/维表变更广播到所有下游实例。
考点：keyBy 是状态局部性前提；Broadcast 状态需注意只读规则侧与内存占用。见 docs/02-time-window、p03 Broadcast 动态 Pattern。

7. 为什么说「反压是信用机制」？和 TCP 背压有何类比？

**参考答案：** 答案：Flink 网络栈用 credit-based flow control：接收端有可用 floating buffer 才发放 credit，发送端无 credit 则停发。类比 TCP 窗口：不是「丢包重试」，而是「窗口为 0 时不发送」。
值班含义：反压指标升高说明下游处理不过来或网络/序列化慢，应沿拓扑向下游找 busy 算子，而不是先加并行度掩盖。见 monitoring 值班五指标与 docs/01-runtime。

8. Managed Memory 主要给谁用？和 Heap 混用的常见误区？

**参考答案：** 答案：Managed Memory 主要服务 RocksDB/状态后端、网络缓冲、批算子内存等，由 Flink 管理；Heap 给用户对象与 JVM。
误区：把大状态放在堆上 ValueState 却指望 RocksDB 自动卸荷——HashMapStateBackend 状态在堆；RocksDBStateBackend 才把 K/V 放到堆外托管内存。调优时先确认 state backend。见 docs/03-state。

9. 如何从 metrics 判断「算子忙」还是「在反压」？

**参考答案：** 答案：看 TM 任务级：`busyTimeMsPerSecond` 接近 1000 表示该任务几乎满负荷计算；`backPressuredTimeMsPerSecond`（及 soft/hard）升高表示因下游信用不足而等待。两者可同时存在（上游算子被下游卡住时 busy 可能下降、backpressured 上升）。
实操：Grafana `flinklab-job-deepdive` 已挂这些序列。见 monitoring/README.md。

10. 什么是 Watermark？它与「事件时间进度」的关系？

**参考答案：** 答案：Watermark(t) 是声明「小于 t 的事件时间数据预期不会再到达」（在允许乱序策略下）。窗口/定时器依据 watermark 触发。
推导：没有 watermark，事件时间窗口永不关；watermark 过快则迟到数据增多。见 docs/02-time-window。

11. Processing Time 与 Event Time 窗口在故障恢复后行为差在哪？

**参考答案：** 答案：Processing Time 依赖机器时钟，恢复后窗口边界与故障前不对齐，结果难复现。Event Time + watermark 与数据时间戳绑定，从 checkpoint 恢复后按事件时间继续，结果可复现（迟到策略仍要配置）。
业务：对账、会话、SLA 监控优先 Event Time。见 docs/02-time-window。

12. 并行度设置的经验法则？何时该加 TM 而非只加 parallelism？

**参考答案：** 答案：先让单并行度链路跑通与正确；再按 CPU 瓶颈升 parallelism；若 Slot/堆/网络打满则加 TM 或提资源。盲目把 parallelism 拉满会导致小文件、checkpoint 放大与调度抖动。
压测：用 benchmark/ 或项目 loadtest 找反压拐点再定并行度。见 benchmark/、best-practice。

13. JobVertex 与 OperatorChain 的关系一句话说明？

**参考答案：** 答案：一个 JobVertex 对应一条可能已链式合并的算子链，在 TM 上通常映射为一个 Task 线程内多算子顺序调用；链内本地调用，链间网络 shuffle。

14. 为什么本地 IDE 跑通不等于 Session 集群跑通？

**参考答案：** 答案：类路径/连接器包、主机名（localhost vs 服务名）、并行度与 Slot、checkpoint 存储权限、时区与 watermark 空闲分区在集群更易暴露。本仓库要求 OrbStack compose 实测，禁止沙箱口头「已跑通」。见 CLAUDE.md。

15. flink run 与 REST 提交的差异？本仓库常用哪种？

**参考答案：** 答案：`flink run` CLI 提交 jar+主类+参数；REST/UI 适合探查。compose 内常用 `docker compose exec jobmanager flink run ...` 或项目 Makefile 封装。生产 K8s 则由 Operator 声明式提交。见 projects/*/Makefile、production/。

16. 什么情况下需要 setMaxParallelism？和 rescale 的关系？

**参考答案：** 答案：maxParallelism 影响 key group 数量，进而限制未来 rescale 上限；默认值通常够用，超大状态/计划扩容时显式设定。Savepoint 兼容与 key group 重分配相关。见 docs/04-checkpoint。

17. Side Output 的典型用途？

**参考答案：** 答案：迟到数据、脏数据、规则旁路、预算熔断旁路（p01 BudgetGate）等需要主流通路与异常通路分离的场景。主输出保持干净契约，旁路可进死信 topic 或降级标签。见 p01、best-practice。

18. DataStream 与 Table/SQL 何时选型？

**参考答案：** 答案：清晰关系型窗口/join/CDC 入湖优先 SQL；复杂 CEP、自定义异步 IO、精细 side output 与状态描述用 DataStream。可混用（同一作业双 API）。见 docs/05-sql、docs/06-table-api。

19. 什么是「管道」与「批」在 Flink 2.x 统一运行时中的体现？

**参考答案：** 答案：同一引擎可跑流与批；批可看作有界流。学习重点仍是事件时间、状态与 checkpoint——批模式可关闭部分流语义。见 docs/00-landscape。

21. Slot 请求失败最常见原因？

**参考答案：** 答案：并行度×资源 > 集群 Slot；或独占 sharing group。处理：降 parallelism、加 TM、检查 setSlotSharingGroup。

22. 什么是 chainable 边？

**参考答案：** 答案：forward 且条件满足时上下游合成一链，本地方法调用代替网络。见算子链条件题。

23. TM 宕机后任务如何恢复？

**参考答案：** 答案：JM 重调度到其他 Slot，从最近 checkpoint 恢复状态与源位点。

24. 为何禁止在 map 里开新线程做阻塞 IO？
