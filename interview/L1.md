1. 一个 Flink 作业从 env.execute() 到 Task 跑起来经历了哪几层图转换？各层分别在哪里生成？

**参考答案：** 答案：典型路径为 StreamGraph → JobGraph → ExecutionGraph → 物理 Task。
1) StreamGraph：客户端/本地环境在 DataStream API 调用时逐步构建（算子、边、并行度提示）。
2) JobGraph：客户端提交前由 StreamGraphGenerator / Optimizer 生成，算子链（chaining）在此折叠，形成 JobVertex。
3) ExecutionGraph：JobManager 收到 JobGraph 后展开为并行 ExecutionVertex/ExecutionEdge，做调度单元。
4) 物理 Task：TaskManager 收到部署后在 Slot 内跑 Task 线程。
考点推导：简历口述应能指出「链合并发生在 JobGraph，并行展开发生在 ExecutionGraph」。见 docs/01-runtime。

2. Slot 与并行度的关系？slot sharing 为什么能让一个 slot 跑整条 pipeline？

**参考答案：** 答案：并行度决定每个算子实例数；Slot 是 TaskManager 提供的资源配额。默认 slot sharing group 下，同一 Job 中不同算子的子任务只要属于同一 sharing group，就可共享一个 Slot（共处一条 pipeline 的上下游）。
理由：Flink 认为同一 Slot 内上下游用本地缓冲交换，节省 Slot 数量——所需 Slot 数 ≈ 最大并行度（在单一 sharing group 时），而非所有算子并行度之和。
反例：把重 CPU 算子单独 setSlotSharingGroup 后，Slot 需求上升；资源不足会部署失败。见 docs/01-runtime。

3. 算子链的合并条件有哪些？什么时候你会手动 disableChaining？

**参考答案：** 答案：合并条件（同时满足）：相同并行度、同一 slot sharing group、前后算子可链式（forward 局部传输）、未被禁用、算子类型允许（如部分 co-process 边界）。
手动 disableChaining / startNewChain 场景：需要独立反压观测、重算子避免与轻算子共线程抢 CPU、或要插入旁路监控。
考点：链式是性能默认优化，但排障时拆链能让 busyTime/backpressure 指标落到具体算子。见 docs/01-runtime。

4. TaskManager 与 JobManager 的职责边界是什么？HA 下 JM 故障作业为何能续跑？

6. Forward / Hash / Rescale / Broadcast 分区各自何时使用？

**参考答案：** 答案：Forward：上下游同并行度本地传递（链内常见）。Hash/KeyBy：按 key 分组保证同 key 进同实例。Rescale：上下游并行度不同时的轮询式局部重平衡。Broadcast：规则/维表变更广播到所有下游实例。
考点：keyBy 是状态局部性前提；Broadcast 状态需注意只读规则侧与内存占用。见 docs/02-time-window、p03 Broadcast 动态 Pattern。

7. 为什么说「反压是信用机制」？和 TCP 背压有何类比？

**参考答案：** 答案：Flink 网络栈用 credit-based flow control：接收端有可用 floating buffer 才发放 credit，发送端无 credit 则停发。类比 TCP 窗口：不是「丢包重试」，而是「窗口为 0 时不发送」。
值班含义：反压指标升高说明下游处理不过来或网络/序列化慢，应沿拓扑向下游找 busy 算子，而不是先加并行度掩盖。见 monitoring 值班五指标与 docs/01-runtime。

9. 如何从 metrics 判断「算子忙」还是「在反压」？
