# L4 · Checkpoint / Savepoint / EOS

> 共 31 题。先自答，再展开「参考答案」。链回 `docs/`/`ai/`/`projects/` 仅作延伸，不替代本题答案正文。

92. Barrier 对齐与非对齐 checkpoint 的时序差异？非对齐代价？

**参考答案：** 答案：对齐：需等所有输入 barrier 到齐再快照，反压/倾斜时耗时长。非对齐：可带着飞行中数据快照，缩短 checkpoint，但恢复时可能有更多重复需 sink 幂等消化，实现更复杂。见 docs/04-checkpoint。

93. 增量 checkpoint 在 RocksDB 上如何实现？删旧 checkpoint 为何不一定省空间？

**参考答案：** 答案：增量上传 SST/变更文件共享；多 checkpoint 引用同一底层文件。删除旧 CP 只减引用，文件可能仍被更新 CP 引用，空间回收滞后于「删除动作」。见 docs/04-checkpoint。

94. Checkpoint 与 Savepoint 的本质区别？

**参考答案：** 答案：Checkpoint：引擎定期容错快照，格式可偏内部优化，保留策略由系统管理。Savepoint：运维触发的、为变更/迁移设计的稳定格式，生命周期由人管理。升级/蓝绿用 savepoint（或 stop-with-savepoint）。见 docs/04-checkpoint、best-practice。

95. 为什么每个有状态算子都要显式 .uid()？不设会发生什么？

**参考答案：** 答案：uid 是状态绑定键。不设则按拓扑遍历自动生成，重构/插算子会变，导致从 savepoint 恢复时状态无法匹配→丢状态或作业拒启。见 best-practice #1。

96. 「checkpoint 成功但数据重复」——哪一环节语义被误解？端到端 EOS 条件？

**参考答案：** 答案：checkpoint 保证 Flink 内部状态与源位置一致，不等于下游外部系统 EOS。端到端需：源可重置、sink 事务/幂等（如 Kafka 事务）、事务超时与 checkpoint 间隔匹配。见 docs/04-checkpoint、docs/07-connectors。

97. Kafka Sink EXACTLY_ONCE 下 transaction.timeout.ms 与 checkpoint 间隔关系？

**参考答案：** 答案：事务超时必须明显大于 checkpoint 间隔+对齐耗时，且 broker `transaction.max.timeout.ms` 放行；否则事务被 abort，表现为丢数/无法提交。见 best-practice #2、e07。

98. unaligned checkpoint 与 buffer debloating 的关系直觉？

**参考答案：** 答案：都为缓解反压下 checkpoint 难完成；debloating 减小飞行缓冲，非对齐减少等待。先治理反压根因，再开高级选项。

99. checkpoint 超时反复发生怎么排查？

**参考答案：** 答案：看 lastCheckpointDuration、反压、状态大小、MinIO/S3 延迟、主线程卡顿（同步 IO）。对齐久→查倾斜；上传久→查远程存储。见 monitoring、benchmark。

100. exactly-once 与 at-least-once 在业务上如何选择？

**参考答案：** 答案：计费/库存要 EOS 或幂等去重；日志类可 at-least-once+下游去重。EOS 成本更高（事务、延迟）。

101. 两阶段提交 sink 的 commit 时机？

**参考答案：** 答案：在 checkpoint 完成回调时 commit 事务，使外部可见与状态快照原子对齐。见 docs/04-checkpoint。

102. savepoint 后能否改并行度？限制是什么？

**参考答案：** 答案：通常可以 rescale，但受 maxParallelism/key group 与算子是否支持；改 uid/拓扑要谨慎。先 staging 验证。

103. Checkpoint 存储选 MinIO 的本仓库动机？

**参考答案：** 答案：本地 S3 兼容、与 OrbStack 一体、不依赖云账号；生产可换真实对象存储端点。见 docker 基座。

104. 外部化 checkpoint 的用途？

**参考答案：** 答案：作业取消后仍保留 CP，便于从 CP 重启（非 savepoint 语义完整替代，但运维上常用）。明确保留策略防盘满。

105. barrier 在算子内如何传递？

**参考答案：** 答案：注入数据流，操作符在对齐点触发 snapshot；广播流等也有对齐语义。理解「barrier 不跃进无界乱序」有助于排障。

106. Changelog / 增量状态后端相关概念（学习向）？

**参考答案：** 答案：目标减少物化成本，把状态变更作日志；需按 Flink 2.2.1 SSOT 文档核对启用条件，禁止未登记特性当默认。见 docs/04-checkpoint。

107. 从 savepoint 启动失败常见原因？

**参考答案：** 答案：uid 不匹配、并行度/maxParallelism 不兼容、缺失状态后端类、jar 与 sp 版本不符、路径权限。对照 bluegreen SOP。见 production/docs/bluegreen-sop.md。

108. Checkpoint 与下游幂等键设计如何配合？

**参考答案：** 答案：即使 EOS sink，对端系统重放仍可能需业务幂等键（订单号）。Flink EOS 不免除业务去重设计。

109. 停机升级：stop-with-savepoint 与 cancel 的差别？

**参考答案：** 答案：前者优雅停并出 savepoint；cancel 可能丢未完成 CP。生产禁止「直接 cancel 当升级」。见 best-practice #8。

110. numberOfFailedCheckpoints 增长说明什么？

**参考答案：** 答案：快照失败（超时、存储错误、任务失败）。与 numRestarts 一起看：是任务在挂还是只是 CP 失败。见 job-deepdive 看板。

111. 本地恢复（local recovery）的收益与前提？

**参考答案：** 答案：TM 重启时尽量用本地状态减少下载；需稳定本地盘与配置开启。容器可写层丢失则回退远程 CP。

112. checkpoint interval 过短的坏处？

**参考答案：** 答案：CPU/网络被快照占满，吞吐下降，事务压力大。

113. tolerable checkpoint failures 如何用？

**参考答案：** 答案：允许偶发失败避免作业立刻 fail；持续失败仍要告警。

114. 从 CP 与从 SP 恢复的话术差异？

**参考答案：** 答案：CP 偏容错自动；SP 偏变更运维。面试强调格式与生命周期。

115. EOS sink 下下游可见延迟来源？

**参考答案：** 答案：要等 CP 完成才 commit，延迟≈对齐+持久化。

116. savepoint 格式跨大版本？

**参考答案：** 答案：需官方兼容矩阵；跨大版本常要中间迁移。本仓库锁 2.2.1。

117. failed checkpoint 与 job fail 的配置桥梁？

**参考答案：** 答案：失败次数阈值；配太严则抖动致命。

118. 对齐时间过长能否只靠加大 timeout？

**参考答案：** 答案：掩盖问题；应治倾斜/反压。timeout 只作保险。

119. Kafka 恰好一次与幂等生产者关系？

**参考答案：** 答案：事务 sink 依赖幂等生产者能力与事务协调；配置要齐全。

120. 增量 CP 与全量 CP 切换影响？

**参考答案：** 答案：存储与恢复时间变化；观察 size 指标。

121. end-to-end 延迟构成？

**参考答案：** 答案：事件产生→源→计算→等 CP 提交→下游可见。

122. 重放源从 earliest 的危险？

**参考答案：** 答案：重复处理历史，压垮下游；用正式位点。
